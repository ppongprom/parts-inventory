# คำสั่งสำหรับ Claude Code ในโปรเจกต์นี้

## ภาพรวมโครงการ

ระบบสต็อกอะไหล่รถสำหรับอู่ซ่อม/ร้านขายอะไหล่ (Next.js + Supabase) — multi-tenant จริง (หลายอู่ใช้
ระบบเดียวกัน แยกข้อมูลด้วย `shop_id` + RLS), มี subscription tier (Trial/Starter/Founder/Pro/
Enterprise), และมี Platform Admin แยกต่างหาก (ทีมงานเจ้าของระบบเอง คนละสิทธิ์คนละบัญชีกับอู่ลูกค้า)

**เอกสารอ้างอิง 3 ไฟล์ อ่านให้ตรงจุดประสงค์:**
- **`SOP.md`** — ขั้นตอนทำงานวันต่อวันของทีมหน้างาน (เจ้าของ/ผู้จัดการ/ช่าง) ต้องทำอะไรก่อน-หลัง
  ต่อฟีเจอร์ ไม่ใช่ spec ระดับโค้ด — เช็คไฟล์นี้ก่อนเวลาจะถามว่า "ฟีเจอร์ X ใช้งานจริงหรือยัง"
- **`README.md`** — บันทึกทางเทคนิคเรียงตามลำดับ session/วันที่ (schema, migration, decision ที่
  เคาะไปแล้ว, bug ที่เจอ+แก้) เช็คไฟล์นี้ก่อนเวลาจะถามว่า "ทำไมถึงออกแบบแบบนี้" หรือหาประวัติว่า
  แก้อะไรไปเมื่อไหร่
- **`USER_MANUAL.md`** — คู่มือผู้ใช้แบบละเอียดราย role (role ไหนทำอะไรได้บ้าง ปุ่มไหนกดแล้วเกิดอะไร)

ทั้ง 3 ไฟล์นี้เป็น source of truth ที่ update ต่อเนื่องทุก session — **ก่อนเริ่มงานใหม่ที่เกี่ยวกับ
ฟีเจอร์ที่มีอยู่แล้ว ให้อ่านส่วนที่เกี่ยวข้องในไฟล์เหล่านี้ก่อนเสมอ** แทนที่จะเดาจากการอ่านโค้ดอย่าง
เดียว เพราะมักมีบริบท/มติ/เหตุผลที่ไม่ได้อยู่ในโค้ดตรงๆ (เช่น ทำไมถึง defer ฟีเจอร์นี้ไว้ก่อน)

## กติกาบังคับ: สรุปแผนก่อนลงมือทำทุกครั้ง

ก่อนเริ่มลงมือแก้โค้ด/schema/migration ที่ไม่ใช่งานเล็กๆ (typo, one-line fix) **ต้องสรุปแผนการทำงาน
พร้อมตัวเลือก/trade-off ให้ผู้ใช้ดูก่อนเสมอ** แล้วรอ confirm ก่อนค่อยเริ่มแก้จริง — ห้ามลงมือทำเลย
โดยไม่ถาม แม้จะดูเป็นทางเลือกที่ชัดเจนอยู่แล้วก็ตาม

ใช้ `EnterPlanMode` เพื่อทำสิ่งนี้ (สำรวจโค้ด → ออกแบบแผน → เขียนแผนลง plan file → `ExitPlanMode`
เพื่อขออนุมัติ) เป็นค่าเริ่มต้นสำหรับงานที่:
- แก้ schema ฐานข้อมูล หรือ RLS policy
- เพิ่ม/แก้ฟีเจอร์ที่กระทบมากกว่า 1-2 ไฟล์
- มีหลายวิธีทำที่เป็นไปได้ (เช่น เลือกระหว่าง grant ตรงๆ กับสร้าง RPC)
- งานที่การ์ด/requirement มีจุดที่ยังไม่ชัดเจนหรือมีข้อสมมติที่ต้องตรวจสอบก่อน

หากพบว่าสมมติฐานของการ์ด/requirement ไม่ตรงกับโค้ดจริง (เช่น อ้างว่ามีฟีเจอร์ X อยู่แล้วแต่จริงๆ
ไม่มี) ให้แจ้งผู้ใช้ก่อนดำเนินการต่อ ไม่ใช่เดาทำไปเอง

## บริบทโปรเจกต์ที่ควรรู้

- Branch หลักที่ทำงานจริง (active dev) คือ `staging` — ไม่ใช่ `main` (`main` = production,
  ตามหลัง `staging` อยู่มาก อย่าตั้งสมมติฐานว่า 2 branch นี้เหมือนกัน)
- Migration ใหม่ทุกไฟล์ต้อง idempotent (`create table if not exists`, เช็ค `pg_constraint`
  ก่อน `add constraint` ฯลฯ) — ดูรายละเอียดกติกาเต็มใน `SOP.md` หัวข้อ "กระบวนการกัน Schema Drift"
- แก้ DB ตรงผ่าน Supabase MCP เมื่อไหร่ ต้อง export กลับเป็นไฟล์ `.sql` ใน `db/` ในเซสชันเดียวกัน
  ห้ามปล่อยค้าง
- ไม่ grant สิทธิ์ตรงบน table/view ให้ `anon`/`authenticated` แบบกว้างๆ ถ้าเลี่ยงได้ — ใช้
  `SECURITY DEFINER` RPC function ที่ scope แคบแทน (กัน Supabase Security Advisor เตือน และ
  ควบคุมสิทธิ์ได้ละเอียดกว่า) ดูตัวอย่างที่ `search_cost_item_history()`/`search_cars()`
- SSH key ของเครื่องนี้ล็อกด้วย passphrase — Claude ไม่มีทาง push ขึ้น GitHub เองได้ ต้องให้ผู้ใช้
  รันคำสั่ง push เองจาก terminal ที่มี credential จริงเสมอ
